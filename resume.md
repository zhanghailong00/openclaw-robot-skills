# 简历项目描述

## 项目名称

基于 AI 智能体的嵌入式机器人自然语言控制系统

## 一句话介绍

在 RK3588 实训箱上部署 OpenClaw AI 智能体，通过飞书实现自然语言控制机械臂、传感器等硬件设备。

## 技术栈

硬件：RK3588 ARM64、4-DOF 机械臂、UVC 摄像头、I2C 传感器
AI：OpenClaw 智能体框架、DeepSeek 大语言模型、YOLOv11 目标检测
通信：UART 串口、GPIO、I2C、WebSocket
语言：Python、Node.js
工具：Linux、systemd、飞书开放平台

## 项目描述

在 RK3588 ARM64 嵌入式平台上部署 OpenClaw AI 智能体框架，将实训箱上的机械臂、摄像头、传感器等硬件封装为 11 个独立 Skill 模块（共 32 个 Python 脚本），通过飞书实现自然语言控制。

主要工作：
- 在 ARM64 平台解决 OpenClaw 兼容性问题（Node.js 内存调优、命令卡死修复）
- 设计 Skill 插件架构，按功能分 5 类（机械臂/视觉/传感器/输出控制/显示），AI 自主编排硬件操作
- 实现视觉引导抓取：YOLO 目标检测 → 像素坐标转换 → 逆运动学解算 → 舵机控制
- 编写合并脚本优化响应速度，从 6-8 秒降至 3-4 秒
- 集成飞书开放平台，通过 WebSocket 实现自然语言交互

## 项目成果

- 实现通过飞书发消息控制机械臂完成抓取、放置、传送带运输等操作
- AI 智能体能自主理解用户意图，自动编排多个硬件模块完成复杂任务
- 11 个 Skill 全部测试通过，飞书端到端测试验证成功
- 编写合并脚本，将多次 Python 启动优化为单次调用，响应速度提升 50%

## 面试话术

### 项目一句话介绍

在 RK3588 嵌入式平台上部署 AI 智能体，让学生通过飞书发消息就能控制机械臂，AI 自己理解任务、自己调用硬件接口。

### 难点 1：ARM64 兼容性问题

OpenClaw 在 RK3588 上运行时，Node.js 命令卡死、内存溢出。通过分析 V8 堆内存机制，用 --max-old-space-size 调优，绕过包装脚本直接调用 Node 运行时，解决了平台兼容性问题。

### 难点 2：Skill 架构设计

最初想把所有硬件控制写成一个大脚本，后来发现这样 AI 无法灵活组合。改为按功能分 5 类 Skill（机械臂、视觉、传感器、输出控制、显示），每个 Skill 独立可调用，AI 自主编排。

### 难点 3：响应速度优化

飞书发指令到机械臂动作需要 6-8 秒，分析发现瓶颈是 Python 每次启动都重新加载模块和连接串口。写了一个合并脚本 grab_and_place.py，将多次调用合并为一次，速度提升 50%。

### 难点 4：视觉引导抓取

实现完整视觉引导链路：YOLO 检测物体 → 像素坐标转换 → 工作台坐标 → 机械臂坐标 → 逆运动学解算 → 舵机控制。每一步都需要标定和校准。

### 项目亮点

- 完整的 AI + 嵌入式 + 机器人系统集成
- 11 个 Skill 模块，32 个脚本，全部测试通过
- 飞书端到端测试验证，AI 能自主理解意图并编排硬件操作
- 从硬件驱动到上层应用的全栈开发能力

### 可能被问到的问题

| 问题 | 回答要点 |
|------|---------|
| 为什么选 OpenClaw 不选 ROS？ | 轻量、教学场景够用、多渠道接入方便 |
| 这个项目能用在工业场景吗？ | 不能，实时性不够，需要 RTOS 或专用控制器 |
| 如果让你重做，你会怎么改？ | 加本地模型（RK3588 NPU）、加实时控制、加安全急停 |
| 机械臂运动学怎么实现的？ | DH 参数法，正运动学算末端位置，逆运动学算关节角度 |
| Skill 架构有什么好处？ | 独立可调用、AI 可灵活组合、易扩展、易维护 |
